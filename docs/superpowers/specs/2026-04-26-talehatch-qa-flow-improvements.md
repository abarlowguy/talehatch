# TaleHatch Q&A Flow Improvements — Design Spec
**Date:** 2026-04-26  
**Status:** Approved for implementation

---

## Overview

Six improvements to TaleHatch covering the Q&A prompt flow, image generation quality, mobile UX, and narrative continuity. Tasks 1–3 all modify the same Q&A prompt screen. Tasks 4–6 are independent.

---

## Task 1 — Final Chapter / End Story Option

### Goal
Allow users to designate any chapter (chapter 2 or later) as the final chapter of their story. The final chapter generates a narrative resolution instead of a cliffhanger.

### Rules
- Chapter 1 always has a cliffhanger. The final chapter option is hidden on chapter 1.
- Minimum story length is 2 chapters — the option first appears on chapter 2.
- No maximum chapter limit.

### UI
A checkbox labelled "Make this the final chapter" appears at the bottom of Question 1 of each chapter (chapter 2+), below the submit/skip row.

When the checkbox is checked, an inline card expands immediately below it containing:
- A label: "FINAL CHAPTER"
- The prompt: "What lesson or message should this story leave with your reader?"
- A text input with placeholder: `e.g. "Being brave means helping others even when it's scary"`
- A "Need ideas?" disclosure (collapsed by default) — when expanded, shows 3 contextual moral suggestions generated by AI based on the story so far (characters, events, themes from prior chapters)
- A "Save →" button to confirm the moral and collapse the card

Tapping a suggestion fills the input (does not auto-submit). The user must tap "Save →" to confirm.

After saving, Q&A continues normally with the remaining prompts. The moral is stored in AppState.

Unchecking the checkbox collapses and discards the moral card, clears `storyMoral`, and sets `isFinalChapter` back to `false`.

### State changes
- Add `isFinalChapter: boolean` to AppState (default `false`)
- Add `storyMoral: string` to AppState (default `""`)
- Persist both fields in the stories DB state JSON

### Chapter generation changes
When `isFinalChapter` is true, the chapter.ts system prompt must:
- Omit the cliffhanger instruction
- Add instruction to write a satisfying narrative resolution
- Include the `storyMoral` value and instruct the AI to weave it naturally into the ending
- Return `CLIFFHANGER: none` so downstream code handles it gracefully

### Moral suggestions API
A new lightweight endpoint (or inline with generate.ts) that accepts `{ story, entities, chapterHistory }` and returns 3 short moral/lesson suggestions contextualised to the story. Uses Claude Haiku.

## Task 2 — Dynamic "Need ideas?" Hints

### Goal
The hint suggestions shown under each Q&A prompt are currently static. They should be generated dynamically based on the current prompt and the answers already given in the session.

### Behaviour
- "Need ideas?" is collapsed by default on every prompt.
- When expanded, 3 suggestions appear. Each suggestion is a short (1–2 sentence) contextual idea tailored to:
  - The specific question being asked
  - The story so far (characters, events, prior answers in this session)
  - The chapter history (prior chapter cliffhangers, key entities)
- Tapping a suggestion fills the text input. Does not auto-submit.

### Implementation
Hints are generated by Claude Haiku via generate.ts (or a new `/api/hints` endpoint). They are fetched lazily — only when the user taps "Need ideas?" for the first time on a given question. A loading spinner shows while fetching. Results are cached in component state so re-opening the disclosure doesn't re-fetch.

The hint prompt receives: current question text, prior answers in this Q&A session, story log, key entities, and recent cliffhanger.

### Moral suggestions (Task 1 overlap)
The "Need ideas?" inside the final chapter moral card follows the same pattern but uses a moral-specific prompt. Can share the same endpoint with a `type: "moral" | "answer"` parameter.

## Task 3 — Skip and Ask Differently

### Goal
Users who can't answer a prompt should be able to skip it or get it rephrased.

### UI (per the approved mockup — Layout C)
- A small "skip" text link appears to the left of the Submit button on every prompt.
- A 🔄 icon appears to the right of the question text.

### Skip behaviour
- Advances to the next prompt without recording an answer.
- The skipped question is not re-asked later in the same session.
- If the AI has enough answers to signal READY, skipping is allowed even mid-flow.

### Ask differently behaviour
- Tapping 🔄 sends the current question back to generate.ts with an instruction to rephrase it.
- The question text updates in place (no page transition). A brief loading state on the 🔄 icon while fetching.
- The rephrased question replaces the original. The user gets one rephrase per question — the 🔄 icon greys out after use.
- Prior answers in the session are passed so the rephrased question stays contextually coherent.

## Task 4 — Fix Input Field Hidden Behind Keyboard (Mobile/iPad)

### Goal
On mobile and iPad, the focused text input must always be visible above the on-screen keyboard.

### Approach
Use the [Visual Viewport API](https://developer.mozilla.org/en-US/docs/Web/API/Visual_Viewport_API) to detect keyboard height and scroll the focused input into view.

On `focusin`, attach a handler that:
1. Reads `window.visualViewport.height` after a short delay (keyboard animation completes ~300ms)
2. Calls `element.scrollIntoView({ behavior: 'smooth', block: 'center' })` on the focused input

On `visualViewport` `resize` event, re-check and re-scroll if the viewport shrinks (keyboard appeared).

Also add `scroll-padding-bottom` to the page container equal to the keyboard height so the browser's native scroll doesn't stop short.

This applies globally to all text inputs in the app, not just Q&A prompts.

## Task 5 — Recency-Weighted Q&A Prompts

### Goal
Q&A questions for chapter N should reference and prioritise events from chapter N-1, with decreasing weight for older chapters.

### Implementation
Pass a `chapterHistory` array to generate.ts's system prompt. Each entry includes:
- `chapterNumber`
- `cliffhanger` (2–3 sentence summary already stored in ChapterRecord)
- `title`

The system prompt instructs Claude Haiku to:
- Prioritise callbacks to the most recent chapter (N-1) when generating questions
- Reference earlier chapters only if directly relevant
- Never ask about events the user hasn't written yet

The cliffhanger summaries already stored in `savedChapters` are sufficient context — no need to pass full chapter text.

### State changes
`savedChapters` is already in AppState and passed to the chapter API. It needs to be passed to generate.ts as well (currently it is not).

## Task 6 — Fix Mid-Chapter Images Losing Story Context

### Goal
Every illustration in a chapter must visually reference the main character and story, regardless of which scene within the chapter it depicts.

### Root cause
`IMAGE_PROMPTS` in chapter.ts are generated from local scene text. The Pollinations AI prompt receives only the scene-level description with the art style suffix — no persistent character or story anchor.

### Fix
Inject a story anchor prefix into every image prompt before it is sent to Pollinations. The anchor includes:
- Main character name and physical description (extracted from `entities` + story log on chapter 1; carried forward in AppState)
- Art style (already available as `artStyle` in AppState)
- Genre/setting (one-line summary, available from chapter 1 genre detection)

Format: `[{characterAnchor}. {settingAnchor}.] {scenePrompt}. {artStyleSuffix}`

The character anchor is built once on chapter 1 and stored in AppState as `characterDescription: string`. It is passed to chapter.ts and prepended to every image prompt before URL construction.

### State changes
- Add `characterDescription: string` to AppState
- Populate it from chapter 1's generation response (add `CHARACTER_DESCRIPTION` to the chapter 1 output format)
- Pass it through to image prompt construction in both chapter.ts and imageRegen.ts

## Shared UI Notes

- All "Need ideas?" disclosures across Tasks 1, 2 use the same collapsed/expanded pattern and the same lazy-fetch + cache behaviour.
- The Q&A prompt component will grow in complexity. Consider extracting it from `app/page.tsx` into a `components/QAPrompt.tsx` component as part of this work.
- All new AppState fields must be added to `INITIAL_STATE`, the serialization path, and the resume/autoSave flow.

## Out of Scope
- Allowing the user to change a chapter back from "final" to non-final after saving (future).
- Multi-moral support (one moral per story).
- Showing the moral text to the reader in the published story (future).
